# MVP Quality Checklist

Usa este checklist para confirmar que el MVP es suficientemente pequeno y construible.

- [ ] Hay un usuario inicial especifico.
- [ ] El problema se puede describir en una frase.
- [ ] La propuesta de valor se entiende sin explicar tecnologia.
- [ ] El MVP tiene una funcionalidad principal.
- [ ] Las funcionalidades post-MVP estan separadas.
- [ ] Hay no objetivos claros.
- [ ] La vertical slice tiene entrada, proceso y salida.
- [ ] La vertical slice puede probarse manualmente.
- [ ] No requiere datos reales sensibles para la primera prueba.
- [ ] No requiere login si no es esencial.
- [ ] No requiere pagos si no valida el problema principal.
- [ ] No requiere integraciones externas en la primera version.
- [ ] Las metricas de exito son observables.
- [ ] La Definition of Done es verificable.

## Señales de alerta

- "Para todos los usuarios".
- "Plataforma completa".
- "IA personalizada entrenada desde cero".
- "Integracion con todo".
- "Despues vemos las metricas".

## Condiciones para aprobar

- Todos los items esenciales de usuario, problema, resultado, no objetivos y vertical slice deben estar marcados.
- Las señales de alerta deben estar ausentes o movidas a post-MVP.
- Si una restriccion impide construir, debe aparecer como riesgo o pregunta bloqueante.

## Preguntas de presion

- Que se puede eliminar sin destruir el aprendizaje principal?
- Que prueba manual demostraria valor en menos de una hora?
- Que decision humana debe ocurrir antes de automatizar?
- Que parte del MVP seria peligrosa si se ejecuta sin revision?
